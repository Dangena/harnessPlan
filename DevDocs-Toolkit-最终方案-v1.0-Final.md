## 0. 文档定位与边界

本文件是 DevDocs Toolkit 的**唯一基线文档**，统一产品定位、架构设计、能力清单与实施路线。

### 明确边界
- **不包含**开发任务派单细节
- **不绑定**特定AI厂商执行分工
- **不覆盖**技术概要设计（HLD）、技术详细设计（LLD）作为工具箱目标产出

---

## 一、项目概述

### 1.1 背景

公司软件开发的前期工作（产品需求、方案设计、投标文档、PPT、UI设计等）目前依赖员工手动基于旧模板编写，效率低、质量参差不齐。需要一个工具箱，让同事们能够按照公司材料模板，结合自身需求，快速完成各类文档的编写与输出。

### 1.2 范围边界

本工具箱聚焦于**软件开发前期中偏业务/产品侧**的文档工作：

| ✅ 在范围内 | ❌ 不在范围内 |
|------------|-------------|
| 调研提纲、访谈纪要 | 技术概要设计（HLD） |
| 需求分析、需求清单 | 技术详细设计（LLD） |
| PRD | 架构设计文档 |
| UI原型 | 接口设计文档 |
| 项目立项方案 | 数据库设计文档 |
| 汇报材料 | 部署方案 |
| 测试用例 | |

> 技术设计类文档属于开发阶段，由技术团队负责。如未来有需要，可作为扩展项单独规划。

### 1.3 核心目标

- 将前期文档产出从"人工拼接"升级为**标准化生产线**
- 让产品经理、项目经理在个人模式下**快速生成高质量文档**
- 建立可治理、可追溯、可扩展的**企业级文档生产底座**

### 1.4 目标用户

| 用户角色 | 典型场景 |
|---------|---------|
| 产品经理 | 需求分析、PRD、UI原型 |
| 项目经理 | 立项方案、汇报材料、工作量估算 |
| Marketing | 汇报PPT、方案包装 |

### 1.5 项目类型匹配

| 项目类型 | 典型特征 | 核心诉求 | 推荐工具 |
|---------|---------|---------|---------|
| **政务IT** | 政策合规强、流程规范、文档格式严格 | 合规检查、格式规范、多版本输出 | ①调研、②需求、③PRD、⑤立项、⑥汇报 |
| **企业IT** | 业务导向、迭代快、成本敏感 | 效率优先、快速迭代、灵活裁剪 | ②需求、③PRD、④UI、⑦测试 |
| **科研** | 创新导向、里程碑明确、周期长 | 技术路线、创新点、成果包装 | ①调研、⑤立项、⑥汇报 |
| **招投标** | 响应招标文件、评分标准明确 | 技术响应得分、文档专业度 | ②需求、③PRD、⑤立项 |
| **内部自建** | 需求明确、团队稳定、持续运维 | 文档完整、测试覆盖、知识沉淀 | ③PRD、⑤立项、⑦测试 |

## 二、总体架构

### 2.1 四层架构

```
┌─────────────────────────────────────────────────────────┐
│                  体验层（Experience）                      │
│    对话入口（自然语言）  |  参数化入口（结构化表单）         │
├─────────────────────────────────────────────────────────┤
│                  编排层（Orchestration）                   │
│    Planner Agent  |  Producer Agent  |  Reviewer Agent   │
├─────────────────────────────────────────────────────────┤
│                  能力层（Skills）                          │
│    core-*  |  survey-*  |  demand-*  |  prd-*  |  ...    │
├─────────────────────────────────────────────────────────┤
│                  治理层（Governance）                      │
│    模板版本治理  |  术语库  |  输出规范  |  指标监控       │
└─────────────────────────────────────────────────────────┘
```

### 2.2 Skill三层依赖关系

```
聚合层（7个工具Skill）  ← 用户直接调用
    ↓  编排调用
领域层（15个专项Skill）  ← 按文档类型/阶段划分
    ↓  复用
基础层（5个核心Skill）   ← 通用能力原子
```

### 2.3 3个核心Agent

| Agent | 职责 | 输入 | 输出 |
|-------|------|------|------|
| **Planner Agent** | 任务识别与参数补全 | 用户目标、项目背景 | 执行计划、缺失信息问询、调用顺序 |
| **Producer Agent** | 文档生成与组装 | 已补全参数 + 模板 + Schema | 目标文档初稿 |
| **Reviewer Agent** | 质量审校与改进建议 | 文档初稿 + 质量规则 | 评分报告、问题列表、修复建议 |

**文档状态流：** `Draft → Review → Approved → Archived`

## 九、Skill vs Agent 形态决策

### 9.1 使用模式

- **Skill形态（80%场景）**：单一文档需求、快速验证、简单串联（≤3步）
- **Agent形态（20%场景）**：完整流程自动化、并行多工具协同、长周期项目

### 9.2 复杂度评分算法

```
得分 = 串联步骤数 × 1.0 + 并行工具数 × 1.5 + 决策分支数 × 2.0 + 上下文长度 × 0.5

≥ 8 → Agent形态
< 8 → Skill形态
```

### 9.3 Agent状态机

| 状态 | 说明 |
|------|------|
| IDLE | 初始状态，等待任务 |
| RUNNING | 任务执行中 |
| WAITING | 等待下游结果/人工确认 |
| COMPLETED | 任务正常完成 |
| FAILED | 任务异常 |
| SUSPENDED | 暂停（等待外部输入） |

## 十三、风险与应对

| 风险 | 应对策略 |
|------|---------|
| 模板频繁变更 | 双周冻结窗口 + 模板管理员 |
| 字段冲突 | 统一Schema为唯一事实源 |
| 跨平台差异 | 仅在安装层适配，不改业务层接口 |
| 可解释性不足 | 输出证据链与可编辑建议 |

## 十四、基线确认清单

在进入《开发计划文档（实施版）》之前，请确认以下5项：

1. **范围确认**：仅覆盖 7 个产出工具 + 2 个治理工具
2. **边界确认**：不纳入技术概要设计、技术详细设计
3. **优先级确认**：第一阶段以"前期需求分析 + PRD"闭环为优先
4. **架构确认**：采用四层架构 + 3 Agent 编排模式
5. **验收确认**：沿用第十二章 KPI 作为阶段验收基准

> 以上5项确认后，再进入下一份《开发计划文档（实施版）》编制。
